Paste @U Base85 serials or deserialized lines (pipes + part tokens like 261, 0, 1, 50| 2, 2730|| …). That text is not the same bytes as WASM @U; the decoder iframe parses it when WASM cannot. YAML/JSON with serial: fields work too. Each row gets part resolution, then checked with the same data-backed rules as Legit Builder (manifest, NCS, inv composition tags from inv_comp_tag_data.js + tag-comp-validation.js, loot schedules, weights, source tables, stats coverage). Parts not on the manifest loot-pool export are listed as FYI only (rewards/vendors/quests are still valid). Drop hints use SOURCE_PATHS_DATA when applicable.

How it works
  • Decode: Same-page WASM + shared decode path as legacy/bl4-bulk-decoder.html (no iframe).
  • Inv tags: Rolling tag pool from comp basetags/addtags, then each part’s dependencytags, exclusiontags, and comp allowlists — same rule set as save-editor.be Legit Item Builder. Missing deps or exclusion conflicts → Fail (data). Raw serial tokens also feed a parallel graph check when slot mapping is ambiguous.
  • Save-editor Legit–style bulk: Same inv exclusion chain + comp slot counts (min/max) as the rolling tag pool in save-editor Legit; not comp allowlist hard-fails (export lists are narrower than real drops), and dependencytags are skipped like editor parity notes in tag-comp-validation.js. NCS “wrong slot” / schedule level / 0% weight are FYI here, not Fail. Use Legit Builder for full strict audit.
  • Loot-pool FYI: off_pool_* rows are reported for visibility only. They do not hard-fail by themselves.
  • Drops: cc-itempool-drop-check.js (legendary shiny pool path).
  • Chunked: Batches of 32; falls back per-serial if a batch times out.
  • Per-part clarity: Legit Builder shows a Selected parts table (drop pool vs inv-tag columns). Here, Notes lists real failure lines (inv tag graph, weights, NCS, etc.) — not the footer stats block.
  • Export: JSON report (KPI + histogram + flat rows), CSV (includes off_pool_summary), or Markdown summary for pasting. Check “Export only rows matching…” to export e.g. uncertain-only after choosing the filter.
  • Alignment flags (align_*): Extra booleans + alignment_summary / alignment_definitions in JSON for comparing row buckets in the same export (e.g. OK vs not Fail vs mapped-only).

Decoder scripts load from assets/js + assets/wasm; keep the project folder layout when hosting.

What do these mean?
Serials
Total unique serials extracted from the input.
Decoded OK
WASM successfully decoded the serial. If this is less than Serials, some input couldn’t be parsed.
OK (data)
Decoded + manifest item found + slot map matched + passes all checks (inv tags, schedules, sources). Strict pass.
Uncertain
Decoded but not strict OK: usually no NCS→manifest slot map (shows “Uncertain (no slot map)”), or warnings from schedules/weights/inv tags (Legit column shows a warn state). The line under the tiles sums buckets so OK + Uncertain + failures = Serials.
Decode failed
WASM couldn’t decode (bad Base85, corrupted, or unknown format).
No manifest
Decoded but manufacturer/item type don’t match any known manifest item.
Fail (data)
Decoded + manifest + mapped but validation failed (inv tags wrong, off loot schedule, etc.). See Notes.
