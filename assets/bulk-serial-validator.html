Bulk serial validator

Paste @U Base85 serials or Gibbed-style deserialized lines (pipes + part tokens like 261, 0, 1, 50| 2, 2730|| …). That text is not the same bytes as WASM @U; the decoder iframe parses it when WASM cannot. YAML/JSON with serial: fields work too. Each row gets part resolution, then checked with the same data-backed rules as Legit Builder (manifest, NCS, inv composition tags from inv_comp_tag_data.js + tag-comp-validation.js, loot schedules, weights, source tables, stats coverage). Parts not on the manifest loot-pool export are listed as FYI only (rewards/vendors/quests are still valid). Drop hints use SOURCE_PATHS_DATA when applicable.

How it works
  • Decode: Same-page WASM + shared decode path as legacy/bl4-bulk-decoder.html (no iframe).
  • Inv tags: Nexus-derived composition basetags and per-slot tag chains — same computeTagCompositionIssues path as Legit Builder. Failures appear in Notes (e.g. lines starting with Inv tags:).
  • Legit: Same stack as Legit Builder — tag-comp-validation.js + inv_comp_tag_data.js + legit-builder-core.js + LegitDecodeHelpers.validateDecodedSerial (no cross-frame access; avoids file:// blocking).
  • Loot-pool FYI: Off-export parts are not treated as uncertain by themselves. Notes / exports still list slot, name, and index under off_pool_* and the FYI line.
  • Drops: cc-itempool-drop-check.js (legendary shiny pool path).
  • Chunked: Batches of 32; falls back per-serial if a batch times out.
  • Per-part clarity: Legit Builder shows a Selected parts table (drop pool vs inv-tag columns). Here, Notes lists real failure lines (inv tag graph, weights, NCS, etc.) — not the footer stats block.
  • Export: JSON report (KPI + histogram + flat rows), CSV (includes off_pool_summary), or Markdown summary for pasting. Check “Export only rows matching…” to export e.g. uncertain-only after choosing the filter.
  • Alignment flags (align_*): Extra booleans + alignment_summary / alignment_definitions in JSON so you can compare strict OK (data) to looser row buckets in the same export (e.g. “not Fail (data)” or ok/warn-only). Definitions are specific to Scooter’s Toolbox / this validator.

Decoder scripts load from assets/js + assets/wasm; keep the project folder layout when hosting.

What do these mean?
Serials
Total unique serials extracted from the input.
Decoded OK
WASM successfully decoded the serial. If this is less than Serials, some input couldn’t be parsed.
OK (data)
Decoded + manifest item found + slot map matched + passes all checks (inv tags, schedules, sources). Strict pass.
Uncertain / map
Decoded but no slot map match, or inv-tag/schedule warnings. Not a hard fail; use Notes to judge.
Decode failed
WASM couldn’t decode (bad Base85, corrupted, or unknown format).
No manifest
Decoded but manufacturer/item type don’t match any known manifest item.
Fail (data)
Decoded + manifest + mapped but validation failed (inv tags wrong, off loot schedule, etc.). See Notes.
